## 1. Summary

Corrected nighttime charging calculations when tomorrow pricing is only available through hourly forecast data.
EVCC now expands hourly forecast entries into 15-minute price points before searching for a charging window.

## 2. Issue Description

When nighttime charging was enabled and `raw_tomorrow` was unavailable, EVCC fell back to `forecast`.
Some Home Assistant pricing integrations expose `forecast` in hourly intervals instead of 15-minute intervals.
EVCC incorrectly reported that no valid charging window existed even though there were valid overnight slots before `finish_by`.

## 3. Root Cause

The pricing normalizer treated `forecast` data as if it already matched the calculator's 15-minute slot granularity.
Hourly forecast entries therefore broke the contiguous-window check used by the scheduling algorithm.

## 4. Resolution

Updated pricing normalization so forecast entries are expanded into four 15-minute slots per hour before the cheapest-window search runs.
Added a regression test covering nighttime charging with forecast-only overnight pricing data.

## 5. Scope and Impact

The fix affects the pricing normalization path used by the charging calculator when `raw_tomorrow` is missing and `forecast` is used instead.
No configuration changes were required.
Residual risk is limited to pricing integrations that provide mixed or irregular interval data outside the documented structures.

## 6. Validation

Validated with automated tests, including a new regression test that reproduces the forecast-only nighttime scheduling scenario.
Full test suite result: `31 passed`.

## 7. Deployment Notes

Deployment is straightforward.
Restart the add-on after upgrading so the updated pricing normalization is loaded.
