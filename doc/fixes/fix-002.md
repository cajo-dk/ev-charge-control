## 1. Summary

Clarified the release workflow so fix records must be created before a fix-level release is pushed and tagged.

## 2. Issue Description

The repository rules required fix records for deployed fixes, but they did not explicitly require the `fix-xxx.md` file to be created before pushing the release commit and creating the release tag.

That left room for a fix record to be added after a release was already tagged, which would mean the fix documentation was not contained in the tagged revision.

## 3. Root Cause

The release workflow in `CONTEXT.md` and `VERSIONING.md` described fix documentation requirements, but it did not define the exact ordering of documentation creation relative to pushing and tagging a fix-level release.

## 4. Resolution

Updated `CONTEXT.md` and `VERSIONING.md` to make the rule explicit: when releasing at a new fix level, the matching `doc/fixes/fix-xxx.md` must be created and completed before the release commit is pushed and before the release tag is created.

## 5. Scope and Impact

This fix affects release governance and documentation workflow only.

Impact:
- Fix documentation is now required to be part of the tagged release contents.
- The release workflow is clearer and less dependent on memory or interpretation.
- No runtime or packaging behavior changed.

Residual risk:
- Existing releases created before this clarification may still have fix documentation created after tagging.
