## 1. Summary

Corrected EVCC startup so retained MQTT control values are restored before the first calculation attempt, preventing false missing-value errors for `current_soc`, `target_soc`, and `finish_by` after restarts.

## 2. Issue Description

After an add-on restart, Home Assistant could still show valid retained values for controls such as `Current SoC`, `Target SoC`, and `Finish By`, while EVCC immediately logged calculation errors that those same values were missing.

That caused the EVCC status entities to flip into an error state on startup even though the broker already held the required retained control state.

## 3. Root Cause

The `2.1.2` restore path subscribed to retained MQTT control state topics, but the scheduler still executed its first calculation immediately after starting the MQTT client.

Because retained MQTT messages arrive asynchronously, EVCC could calculate against an empty runtime store before the broker had replayed values such as `current_soc`, `target_soc`, and `finish_by`.

## 4. Resolution

Updated startup sequencing so EVCC now:

- waits for MQTT connection confirmation before the initial calculation;
- waits briefly for retained control-state replay to populate required calculation inputs; and
- proceeds normally after a bounded timeout when retained values are genuinely unavailable.

Added regression tests covering startup restore of retained control values and timeout behavior when values are absent.

## 5. Scope and Impact

This fix affects startup initialization only.

Changed areas:

- scheduler startup sequencing
- MQTT publisher connection coordination
- regression tests for retained-state startup restore

No user configuration changes are required.
