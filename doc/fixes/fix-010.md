## 1. Summary

Changed EVCC's no-schedule placeholder for calculated start and end times from blank strings to `--:--`.
Kept helper synchronization and runtime schedule evaluation aligned with the new placeholder.

## 2. Issue Description

When EVCC had no estimated start or end available, the `start` and `end` fields were emitted as blank strings.
That made the output less clear in Home Assistant and did not match the desired user-facing placeholder format.

The same blank representation also propagated into the optional start and end `input_text` helpers introduced in the previous release.

## 3. Root Cause

The calculator's placeholder result used empty strings for `start` and `end`.
Runtime helper synchronization and schedule-detection logic inherited that representation directly.

## 4. Resolution

Updated the calculator placeholder output to emit `--:--` for both `start` and `end` when no schedule is available.
Adjusted runtime helper synchronization so configured `input_text` helpers also receive `--:--`.
Updated the schedule-detection guards so `--:--` is treated as "no schedule" and is not parsed as an executable time.

## 5. Scope and Impact

The fix affects the representation of missing schedule times in MQTT output, runtime payload handling, and optional start/end helpers.
Calculated schedules with real times are unchanged.
Users should now see `--:--` consistently wherever EVCC exposes unavailable start or end times.
