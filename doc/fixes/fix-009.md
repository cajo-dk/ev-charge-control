## 1. Summary

Stopped EVCC from repeatedly issuing charger shutoff commands after the target SoC has already been reached.
Reduced scheduler log noise by downgrading the recurring next-calculation heartbeat from `INFO` to `DEBUG`.

## 2. Issue Description

When the EV reached the configured target SoC, the state machine correctly disabled the charger.
However, on later minute ticks EVCC could issue the same shutoff command again if Home Assistant still reported the charger as enabled.
That caused repeated downstream automations and alert scripts to fire even though EVCC had already made the required state transition.

Separately, the scheduler emitted a routine "next calculation scheduled" message at `INFO` level on every loop iteration, which added noise to normal operational logs.

## 3. Root Cause

The state machine was evaluated as a pure function of the current observed state on every minute tick.
For the unconditional `SoC = Reached` rule, that meant EVCC would keep commanding `set_charging = No` as long as the observed state still matched the rule.
The control loop did not retain enough runtime context to recognize that the auto-reset action had already been sent on a previous tick.

The scheduler heartbeat was logged at `INFO` even though it is routine diagnostic detail rather than an operator-significant event.

## 4. Resolution

Extended the runtime output payload to retain the most recent state-machine rule and used that to suppress repeated actions when the same `auto_reset_soc_reached` rule is encountered on consecutive ticks.
Added regression coverage for the case where Home Assistant continues to report the charger as on after EVCC sends the initial turn-off command.
Changed the scheduler heartbeat log message to `DEBUG`.

## 5. Scope and Impact

The fix affects only the runtime execution loop and logging behavior.
Charging still stops when target SoC is reached, but EVCC now issues that command only once per continuous auto-reset condition.
Users with alerting automations tied to charger switch operations should no longer receive repeated notifications from EVCC for the same target-SoC event.
