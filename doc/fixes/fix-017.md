## 1. Summary

Persisted EVCC-owned control values locally so restarts no longer depend on MQTT retained-state replay to restore required startup inputs such as `target_soc`, `current_soc`, and `finish_by`.

## 2. Issue Description

Even after adding a bounded startup wait in `2.1.3`, some environments still restarted into calculation errors such as `Missing value for 'target_soc'.`

The logs showed EVCC connecting to MQTT successfully and waiting for retained control-state replay, but the replay never supplied several required control values before the first calculation.

## 3. Root Cause

Home Assistant could still display the last known MQTT entity state from its own state restoration, while the MQTT broker did not necessarily replay retained control-state messages for all EVCC inputs during add-on startup.

EVCC therefore still had a gap: it depended on external replay for control-state recovery even though those values are logically owned by EVCC and should survive restarts independently.

## 4. Resolution

Updated EVCC to persist its own MQTT control values in a local runtime-state file under `/data` and restore them on startup before MQTT synchronization begins.

This keeps EVCC’s required startup inputs available across restarts even when retained MQTT replay is incomplete, while still allowing later MQTT control updates to override the restored values.

## 5. Scope and Impact

This fix affects runtime initialization and control-state durability.

Changed areas:

- local persistence of EVCC-owned control values
- startup restore path for required runtime inputs
- regression tests covering persisted control-state restore

No user configuration changes are required.

After release, the next successful runtime update will repopulate EVCC’s local control-state file. Future restarts will restore those values even if MQTT retained replay is incomplete.
