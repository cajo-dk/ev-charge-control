## 1. Summary

Restored missing EVCC control values from Home Assistant’s own MQTT entity states during startup so EVCC can recover inputs such as `target_soc`, `battery_capacity`, and `charger_speed` even when the MQTT broker does not replay retained control-state messages.

## 2. Issue Description

In some environments, Home Assistant still displayed the last known EVCC control values after restart, but EVCC restarted into `Missing value for 'target_soc'.` and similar startup calculation failures.

This still happened after adding MQTT replay waiting and local persistence because EVCC could start without a complete broker replay and without having yet written a local runtime-state file containing every required control.

## 3. Root Cause

Home Assistant restores MQTT entity state independently of the broker. That means the EVCC control entities visible in Home Assistant can hold valid values even when the broker does not replay those same values back to EVCC at startup.

EVCC did not previously use those Home Assistant entity states as a recovery source for its own missing control values.

## 4. Resolution

Updated EVCC startup so that after MQTT restore is attempted, it queries Home Assistant for the state of its own control entities and fills any still-missing startup control values from those entities.

This keeps MQTT and local persistence as the primary restore paths while using Home Assistant’s restored EVCC entity state as a final startup fallback.

## 5. Scope and Impact

This fix affects startup initialization only.

Changed areas:

- startup control-value recovery
- Home Assistant fallback for EVCC-owned controls
- regression tests for Home Assistant fallback restore

No user configuration changes are required.
