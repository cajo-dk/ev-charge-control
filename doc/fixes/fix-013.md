## 1. Summary

Clarified the deployment workflow so releases can only be created from `main`, with release-bound `fr-xxx` and `fix-xxx` branches merged first and removed afterward.

## 2. Issue Description

The repository still contained old `fr-` and `fix-` branches even after their changes had already been incorporated into `main`.
The documented workflow did not state the release-branch expectations clearly enough, which left room for inconsistent release preparation and branch cleanup.

## 3. Root Cause

The deployment rules documented branching for feature and fix work, but they did not explicitly require that releases be cut only from `main`, that all release-bound topic branches be merged before release, or that merged topic branches be removed afterward.

## 4. Resolution

Updated `CONTEXT.md` and `VERSIONING.md` to make the release workflow explicit.
The revised rules require releases to be created only from `main`, require release-bound `fr-xxx` and `fix-xxx` branches to be merged into `main` before release, and direct maintainers to remove those branches after merging.

## 5. Scope and Impact

This fix affects repository governance and deployment workflow documentation only.
It does not change runtime behavior, configuration, or packaging behavior of EVCC.
It reduces the chance of future branch sprawl and inconsistent release preparation.

`VERSIONING.md`
