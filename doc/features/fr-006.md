## 0. Feature Request

- REQUEST ID: fr-006
- DATE: 2026.03.15
- STATUS: Implemented
- RELATED VERSION: 1.6.0
- IMPLEMENTATION BRANCH: fr-006

## 1. Summary

Expose the EV SoC captured at charge start through a configurable Home Assistant `input_number` helper.
This allows automations, dashboards, and users to consume the charge-start SoC directly without having to parse EVCC's MQTT attributes.

## 2. Problem Statement

EVCC already tracks `soc_at_charge_start` internally and publishes it as part of its MQTT output payload.
That is not convenient for Home Assistant users who want to reference the charge-start SoC in simple automations, helpers, or UI cards built around native helper entities.

The lack of a dedicated helper means downstream consumers must either parse the MQTT payload or reconstruct the value indirectly.
Providing a direct `input_number` output aligns this runtime value with the rest of the Home Assistant helper workflow already used by EVCC users.

## 3. Scope

### 3.1. Functional Requirements

- Add a new optional configuration field for an `input_number` entity that EVCC will use to publish `soc_at_charge_start`.
- When EVCC has a known charge-start SoC, write that numeric value to the configured `input_number` helper.
- When EVCC does not have a known charge-start SoC, reset the configured helper to `0`.
- Keep the helper synchronized with EVCC runtime updates, including startup, charge start, charge stop, and target-SoC reset scenarios.
- Preserve the existing MQTT output field for `soc_at_charge_start`.
- If the helper is not configured, EVCC must continue operating without error.

### 3.2. Non-functional Requirements

- The implementation must not introduce repeated helper writes on every minute tick when the value has not changed.
- Logging should remain concise and should only emit helper-write messages when useful for diagnosing behavior.
- The change must remain compatible with Home Assistant service-call patterns already used by EVCC.

### 3.3. Assumptions and Dependencies

- Home Assistant provides a valid `input_number` helper entity whose range can accommodate SoC values from `0` to `100`.
- EVCC will use the Home Assistant API to update the helper rather than writing through MQTT.
- Home Assistant `input_number` does not support an empty value, so EVCC uses `0` as the explicit reset value when no charge-start SoC is available.

## 4. Out of Scope

- Replacing the MQTT runtime output with helper-only output.
- Persisting charge-start SoC across restarts beyond the existing EVCC runtime behavior.
- Adding additional helper outputs for other runtime fields in this feature.
- Creating or managing the `input_number` helper automatically in Home Assistant.

## 5. Configuration and Data Impact

- Add a new optional config field for the `input_number` helper entity ID.
- Update add-on configuration metadata and user-facing translations for the new field.
- No database or file migration is expected.

## 6. Acceptance Criteria

- EVCC configuration supports an optional `input_number` entity for `soc_at_charge_start`.
- When charging starts, EVCC writes the current SoC to the configured helper.
- While the captured charge-start SoC remains unchanged, EVCC does not repeatedly write the same helper value every minute.
- When EVCC no longer has a valid charge-start SoC value, EVCC resets the helper to `0`.
- Existing MQTT output continues to include `soc_at_charge_start`.
- Automated tests cover helper writes, unchanged-value suppression, and reset behavior.

## 7. Implementation Notes

- The current runtime flow already computes `soc_at_charge_start` in `process_minute_tick()` and includes it in the output payload, so the helper sync should likely be attached near the final runtime-output path to keep behavior centralized.
- The Home Assistant API client currently exposes switch and input_boolean service helpers; it will need an `input_number` update capability.
- EVCC resets the helper to `0` when no charge-start SoC is available because `input_number` cannot represent an empty value.

## 8. Test Plan

- Add unit tests for the Home Assistant API client service call used to update the `input_number` helper.
- Add application tests covering:
  - writing the charge-start SoC when charging begins;
  - not rewriting the helper when the value is unchanged on later ticks;
  - resetting the helper to `0` when the runtime value becomes unavailable;
  - skipping helper writes when the helper is not configured.

## 9. Manual QA Checklist

- Configure an `input_number` helper in Home Assistant and point EVCC at it.
- Start a charging session and confirm the helper updates to the EV SoC at the moment charging starts.
- Let EVCC continue running for multiple minute ticks and confirm the helper is not rewritten unnecessarily.
- End charging or reach target SoC and confirm the helper resets to `0` when the charge-start SoC is no longer available.

## 10. Deployment and Rollback Notes

- Users who want the feature must create and configure the `input_number` helper before enabling the new setting.
- Rollback is straightforward: remove the configuration value and redeploy the previous version.
