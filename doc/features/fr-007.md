## 0. Feature Request

- REQUEST ID: fr-007
- DATE: 2026.03.15
- STATUS: Implemented
- RELATED VERSION: 1.7.0
- IMPLEMENTATION BRANCH: fr-007

## 1. Summary

Expose EVCC's calculated charging start time and calculated charging end time through optional Home Assistant `input_text` helpers.
This allows automations, dashboards, and template users to consume the schedule boundaries directly without parsing MQTT attributes.

## 2. Problem Statement

EVCC already calculates and publishes the charging `start` and `end` times through its MQTT output payload.
That works for MQTT-based consumers, but it is inconvenient for Home Assistant users who want to use the values in native helpers, automations, or UI cards built around standard helper entities.

Without dedicated helpers, downstream consumers must parse MQTT state or attributes to extract these times.
That adds unnecessary complexity for simple automations such as checking whether charging is scheduled to start at a specific time or displaying the planned charging window in the UI.

## 3. Scope

### 3.1. Functional Requirements

- Add an optional configuration field for a Home Assistant `input_text` helper that will receive the calculated start time.
- Add an optional configuration field for a Home Assistant `input_text` helper that will receive the calculated end time.
- When EVCC has calculated `start` and `end` values, write them to the configured helpers.
- When EVCC has no calculated `start` or `end` value, write a deterministic empty representation to the corresponding helper.
- Keep the helper values synchronized with EVCC's latest runtime output, including startup, recalculation, and error-result scenarios.
- Preserve the existing MQTT output fields for `start` and `end`.
- If one or both helpers are not configured, EVCC must continue operating without error.

### 3.2. Non-functional Requirements

- EVCC must not repeatedly rewrite the same helper value on every tick when the calculated value has not changed.
- Logging should remain concise and should not add noisy helper-write logs during normal operation.
- The feature must reuse existing Home Assistant API patterns already present in EVCC.

### 3.3. Assumptions and Dependencies

- Users who want this feature will create `input_text` helpers in Home Assistant and configure their entity IDs in EVCC.
- EVCC will update the helpers through the Home Assistant API, not through MQTT.
- Home Assistant `input_text` helpers can represent the current EVCC schedule values as strings such as `02:15`, `05:30`, or `--:--`.

## 4. Out of Scope

- Replacing MQTT as the primary EVCC output path.
- Changing the time format currently used in EVCC output.
- Writing additional schedule fields such as `complete_by` or status to helpers as part of this feature.
- Automatically creating Home Assistant helpers for the user.

## 5. Configuration and Data Impact

- Add two new optional config fields for the start-time and end-time helper entity IDs.
- Update add-on configuration metadata and translations for the new settings.
- No persistent data migration is expected.

## 6. Acceptance Criteria

- EVCC configuration supports optional `input_text` helpers for calculated start and calculated end.
- When EVCC calculates a start time, the configured start helper is updated with that value.
- When EVCC calculates an end time, the configured end helper is updated with that value.
- When EVCC has no schedule value for start or end, the corresponding helper is set to the documented empty representation.
- EVCC does not repeatedly write unchanged helper values on later ticks.
- Existing MQTT output still contains `start` and `end`.
- Automated tests cover helper writes, unchanged-value suppression, empty-value handling, and missing-helper configuration.

## 7. Implementation Notes

- EVCC already produces `start` and `end` in its output payload, so helper synchronization should likely attach near the final payload-writing path rather than duplicating logic in the calculator.
- The Home Assistant API client already supports `input_text.set_value`, which can likely be reused for this feature.
- Empty schedule values should stay aligned with the current EVCC output convention, which uses `--:--` when no value is available.

## 8. Test Plan

- Add application tests covering:
  - writing the calculated start helper;
  - writing the calculated end helper;
  - skipping writes when the helper value is already current;
  - writing empty strings when start or end is unavailable;
  - skipping helper sync when the config fields are unset.

## 9. Manual QA Checklist

- Configure one or both `input_text` helpers in Home Assistant and point EVCC at them.
- Trigger a calculation that produces a valid charging window and confirm the helpers show the expected `HH:MM` values.
- Trigger a recalculation without a schedule and confirm the helpers show the documented empty representation.
- Let EVCC continue running for multiple ticks and confirm unchanged helper values are not rewritten unnecessarily.

## 10. Deployment and Rollback Notes

- Users who want the feature must create the `input_text` helpers before enabling the new settings.
- Rollback is straightforward: clear the helper entity IDs from configuration and redeploy the previous version.
