## 0. Feature Request

- REQUEST ID: fr-010
- DATE: 2026.04.14
- STATUS: Deployed
- RELATED VERSION: 2.2.0
- IMPLEMENTATION BRANCH: fr-010

## 1. Summary

Refine EVCC manual charging control so the current `Start` action becomes a `Start/Stop` override toggle, so disabling automatic charging fully disengages scheduling, and so users can optionally keep charger power engaged after the vehicle reaches target SoC to support cabin preheating or similar behavior.

## 2. Problem Statement

The current EVCC runtime mixes automatic scheduling with limited manual intervention.

Today, the `Start` button is one-way only, schedule authorization being turned off does not explicitly define a fully disengaged scheduler mode, and there is no explicit option to keep charger power engaged after target SoC is reached.

That creates three UX and runtime gaps:

- manual override cannot be used symmetrically to start and stop charging;
- users cannot clearly put EVCC into a completely hands-off mode without leaving stale calculated schedules visible; and
- vehicles that need charger power to remain available after charging finishes cannot be supported cleanly.

## 3. Scope

### 3.1. Functional Requirements

- Replace the current `Start` button behavior with a `Start/Stop` toggle.
- When the toggle is turned on while EVCC is waiting for a future scheduled session:
  - EVCC must start charging immediately; and
  - EVCC must de-authorize automatic charging so the manual override disengages the automatic schedule.
- When the toggle is turned on while charging is not active and there is no waiting schedule, EVCC must treat the action as a manual start request.
- When the toggle is turned off while charging is active, EVCC must stop the current charge session.
- When the `schedule_authorized` control is turned off:
  - EVCC must fully disengage automatic scheduling and no longer interfere with charger state;
  - EVCC must clear any calculated schedule output, including calculated start and calculated end.
- When `schedule_authorized` is turned on while no car is connected, or when the car becomes disconnected:
  - EVCC must clear the current schedule;
  - EVCC must not calculate a new schedule until the cable is plugged in again.
- Add a new `continuous_power` toggle control.
- When `continuous_power` is on:
  - EVCC must keep the charger engaged after the vehicle reaches target SoC if the vehicle remains connected.
- When `continuous_power` is off:
  - EVCC must end the charge session when target SoC is reached, consistent with current stop behavior.

### 3.2. Non-functional Requirements

- Manual override behavior must be deterministic and easy to reason about from the MQTT/Home Assistant control surface.
- Automatic and manual modes must not conflict silently; the active mode should be reflected clearly in state and status outputs.
- Clearing schedules must not leave stale calculated times visible in MQTT sensors or aggregate attributes.

### 3.3. Assumptions and Dependencies

- The current `Start` action may need to change from an MQTT `button` to a persistent MQTT `switch` or similar toggle-capable entity.
- The existing `schedule_authorized` control remains part of the runtime model and continues to represent whether automatic scheduling is allowed.
- `continuous_power` is intended to keep the charger switch engaged after charge completion for connected-vehicle auxiliary functions such as preheating, not to force recalculation or exceed target SoC.

## 4. Out of Scope

- Battery conditioning strategies beyond keeping charger power available after charging completes.
- Vehicle-specific integrations for HVAC, preconditioning, or departure planning.
- Redesign of pricing logic or Home Assistant source selection outside the schedule-clearing behaviors listed above.

## 5. Configuration and Data Impact

- Add a new MQTT/Home Assistant-exposed control for `continuous_power`.
- Replace the existing `Start` action discovery model with a toggle-capable control model.
- Update runtime state handling so schedule-clearing can explicitly blank calculated start and end outputs when automation is disengaged or the cable is disconnected.
- Update documentation and discovery payloads to explain the difference between:
  - manual start/stop override;
  - automatic scheduling authorization; and
  - continuous charger power after target SoC.

## 6. Acceptance Criteria

- Users can toggle manual charging on and off through the EVCC control surface.
- Turning manual charging on while waiting for a future schedule starts charging immediately and disables automatic scheduling.
- Turning manual charging off while charging is active stops the charge session.
- Turning `schedule_authorized` off fully disengages EVCC scheduling logic and clears calculated schedule outputs.
- If no car is connected, enabling automatic charging does not produce or retain an active schedule until the cable is plugged in.
- Disconnecting the cable clears the active schedule.
- A `continuous_power` toggle is available on the EVCC device.
- With `continuous_power=off`, EVCC stops charging at target SoC.
- With `continuous_power=on`, EVCC keeps charger power engaged after target SoC is reached while the car remains connected.
- Automated tests cover manual toggle transitions, schedule clearing, disconnect handling, and continuous-power behavior.

## 7. Implementation Notes

- This FR likely requires state-machine changes, not only UI/discovery changes.
- The current `Start` path is edge-triggered and one-shot; implementing a true toggle will require explicit runtime state for manual override.
- The runtime should distinguish between:
  - automatic scheduling active;
  - manual charging override active; and
  - charger hold-active because `continuous_power` is enabled after target SoC.
- Clearing schedules should update both the dedicated MQTT schedule sensors and the aggregate payload consistently.
- `continuous_power` is the intended internal field name; user-facing documentation can label it `Continuous Power`.

## 8. Test Plan

- Add tests for replacing the current `Start` action with a toggle-capable control.
- Add tests proving manual-on starts charging immediately when a future schedule exists and disables automatic scheduling.
- Add tests proving manual-off stops an active session.
- Add tests proving `schedule_authorized=off` clears calculated start and end outputs and blocks scheduler interference.
- Add tests for enabling automatic charging while unplugged and for unplugging during an authorized state, both of which should clear schedule outputs.
- Add tests for `continuous_power=off` stopping charging at target SoC.
- Add tests for `continuous_power=on` keeping charger engagement active after target SoC while connected.
- Add tests for MQTT discovery payload changes and aggregate payload consistency.

## 9. Manual QA Checklist

- Toggle manual charging on while EVCC is waiting for a scheduled session and confirm charging starts immediately and automatic authorization is turned off.
- Toggle manual charging off while charging is active and confirm EVCC stops the charger.
- Turn `schedule_authorized` off and confirm calculated start/end are cleared and EVCC does not interfere with charger state.
- Turn `schedule_authorized` on while unplugged and confirm no schedule is retained or recalculated until the cable is plugged in.
- Enable `Continuous Power`, charge to target SoC, and confirm charger engagement remains on while connected.
- Disable `Continuous Power`, charge to target SoC, and confirm the session ends normally.

## 10. Deployment and Rollback Notes

- This FR changes the operator-facing charging-control model and should be called out clearly in release notes.
- Existing automations or dashboards that assume `Start` is a stateless button will need updating if the entity type changes.
- Rollback would need to restore the one-shot `Start` action model and remove the `continuous_power` control.
