## 0. Feature Request

- REQUEST ID: fr-001
- DATE: 2026.03.14
- STATUS: Released
- RELATED VERSION: 1.1.0
- IMPLEMENTATION BRANCH: fr-001

## 1. Summary

Add a Home Assistant `input_boolean` helper configuration named `Nighttime Charging Only`.
When the helper is on, EV Charge Control may only schedule charging between the next midnight and the configured finish-by time.
When the helper is off, EV Charge Control keeps the existing behavior and may schedule charging from the current time to the finish-by time.

## 2. Problem Statement

The current calculator always searches from the next available 15-minute slot onward.
That does not support users who want overnight-only charging while still using EVCC to find the cheapest valid continuous window before the deadline.

## 3. Scope

### 3.1. Functional Requirements

- Add a required add-on configuration field that stores the entity ID of a Home Assistant `input_boolean` helper.
- Read the helper state on each calculation cycle through the existing Home Assistant API integration.
- Interpret `on` as enabling nighttime-only scheduling.
- Interpret `off` as disabling nighttime-only scheduling.
- When enabled, restrict the earliest possible charging start to the next midnight in the runtime timezone.
- When disabled, keep the current earliest-start behavior based on the next 15-minute boundary after the calculation time.
- Return an error result if the helper is missing, unavailable, or not in `on` or `off` state.
- Return an error result if no valid charging window exists within the nighttime-only restriction.

### 3.2. Non-functional Requirements

- Keep pricing attribute logging at `DEBUG` only.
- Include the helper state in the normal live-input summary logging.
- Preserve the existing result payload structure written to Home Assistant.

### 3.3. Assumptions and Dependencies

- The helper is implemented as a standard Home Assistant `input_boolean`.
- Runtime timezone handling continues to use the add-on container's local timezone.
- The existing finish-by interpretation remains unchanged.

## 4. Out of Scope

- Splitting charging into multiple disjoint windows.
- Adding daytime-only or manual override behavior.
- Changing the result payload schema.

## 5. Configuration and Data Impact

- Add a new required add-on option: `nighttime_charging_only_entity`.
- No data migration is required.

## 6. Acceptance Criteria

- The add-on configuration requires an entity ID for the Nighttime Charging Only helper.
- The helper state is fetched on each calculation cycle.
- When the helper is off, the calculator behaves as before.
- When the helper is on, the calculator never schedules a start earlier than the next midnight.
- If no valid nighttime-only window exists before finish-by, EVCC writes an error result instead of falling back.
- Automated tests cover the new helper parsing and scheduling behavior.

## 7. Implementation Notes

- The feature should be implemented by extending the existing runtime input model rather than adding a separate calculator path.
- Earliest-start selection is the only scheduling rule that changes for this FR.

## 8. Test Plan

- Add config validation coverage for the new required helper entity.
- Add runtime tests for helper-controlled earliest-start behavior.
- Add runtime coverage for invalid or impossible nighttime-only scheduling cases.
- Keep the existing scheduler tests unchanged.

## 9. Manual QA Checklist

- Configure a valid `input_boolean` helper in Home Assistant.
- Verify that `off` allows scheduling before midnight when cheaper slots exist.
- Verify that `on` prevents scheduling before midnight.
- Verify that an invalid helper state produces an error payload.

## 10. Deployment and Rollback Notes

- Update add-on configuration in Home Assistant before enabling the feature in production.
- If rollback is required, revert to the previous release and remove the new option from the add-on configuration.
