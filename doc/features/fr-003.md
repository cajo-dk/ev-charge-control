## 0. Feature Request

- REQUEST ID: fr-003
- DATE: 2026.03.14
- STATUS: Released
- RELATED VERSION: 1.3.0
- IMPLEMENTATION BRANCH: fr-003

## 1. Summary

Extend the EVCC result payload so Home Assistant can see both the calculated schedule and the current execution context in one output object.
The new output fields must expose the configured completion deadline, charger execution state, authorization state, and relevant SoC values.

## 2. Problem Statement

The current output only exposes `start`, `end`, `timestamp`, and `status`.
That is not sufficient to understand whether the charger is currently enabled, whether execution is still authorized, what SoC the car had when charging began, or how the current SoC compares with the target SoC.
Users must inspect several other entities to understand the live charging state.

## 3. Scope

### 3.1. Functional Requirements

- Extend the EVCC output JSON written to the configured `input_text` helper.
- Preserve the existing output fields:
  - `start`
  - `end`
  - `timestamp`
  - `status`
- Add these new output fields:
  - `complete_by`
  - `authorization_enabled`
  - `charger_enabled`
  - `soc_at_charge_start`
  - `current_soc`
  - `target_soc`
- `complete_by` must reflect the configured finish-by deadline for the active calculation cycle.
- `authorization_enabled` must reflect the current state of the configured schedule authorization helper.
- `charger_enabled` must reflect the current state of the configured charger control switch.
- `soc_at_charge_start` must be set when EVCC starts charging and must remain unchanged during the active charging session.
- `current_soc` must reflect the EV's current SoC at the time the output is written.
- `target_soc` must reflect the configured target SoC at the time the output is written.
- When charging has not yet started, `soc_at_charge_start` should be blank.
- Error payloads must still be written when EVCC encounters invalid runtime input, and the new fields should be populated when practical from known state.

### 3.2. Non-functional Requirements

- Keep the payload JSON small enough to remain suitable for storage in the configured Home Assistant `input_text` helper.
- Keep the existing scheduler and charging behavior unchanged except for the expanded output content.
- Ensure the new fields are updated consistently on both calculation writes and execution-state writes.

### 3.3. Assumptions and Dependencies

- `complete_by` means the configured finish-by deadline, not the calculated charging end time, because `end` already exists in the payload.
- `soc_at_charge_start` can be held in EVCC runtime state and rewritten into the output while the add-on process remains running.
- The current result helper remains the single published output contract for Home Assistant consumers.

## 4. Out of Scope

- Persisting charge-start SoC across add-on restarts.
- Changing the helper type away from `input_text`.
- Creating additional Home Assistant entities for status reporting.

## 5. Configuration and Data Impact

- No new configuration fields are required.
- The structure of the JSON stored in the configured output helper will change.
- Any external consumer of the result payload must tolerate the additional keys.

## 6. Acceptance Criteria

- The output payload includes the six new fields on successful writes.
- `complete_by` matches the active finish-by deadline.
- `authorization_enabled` and `charger_enabled` match the current Home Assistant entity states.
- `soc_at_charge_start` is captured when EVCC turns charging on and remains stable during the active charging session.
- `current_soc` and `target_soc` reflect the latest values read by EVCC.
- Existing fields remain present and backward-compatible.

## 7. Implementation Notes

- The output builder should likely be centralized so both calculation writes and execution-related writes produce the same schema.
- Runtime state will be needed for `soc_at_charge_start` unless the value is persisted elsewhere.
- The feature should avoid duplicating payload-construction logic across `runtime.py` and `app.py`.

## 8. Test Plan

- Add payload-shape tests covering successful calculation output.
- Add execution-path tests covering authorization and charger-state fields.
- Add tests proving `soc_at_charge_start` is set when charging starts and remains unchanged during the locked charging period.
- Add error-path tests to confirm the new payload shape is preserved as far as practical.

## 9. Manual QA Checklist

- Confirm the result helper JSON now includes the new fields.
- Start a scheduled charge and verify `soc_at_charge_start` is captured at the moment EVCC enables the charger.
- Verify `authorization_enabled` turns false when EVCC starts charging.
- Verify `charger_enabled`, `current_soc`, and `target_soc` reflect live state.

## 10. Deployment and Rollback Notes

- Home Assistant dashboards, templates, or automations that parse the EVCC result JSON can start using the new fields after deployment.
- Rollback is straightforward because the existing keys remain in the payload.
