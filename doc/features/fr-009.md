## 0. Feature Request

- REQUEST ID: fr-009
- DATE: 2026.04.13
- STATUS: Implemented
- RELATED VERSION: 2.1.0
- IMPLEMENTATION BRANCH: fr-009

## 1. Summary

Refine the Release `2.x.x` EVCC configuration experience so key numeric values are entered as typed inputs instead of sliders, and so pricing plus charger integration can be selected directly from Home Assistant entities rather than mirrored manually through external MQTT automations.

## 2. Problem Statement

Release `2.0.0` moved EVCC to an MQTT-owned runtime model, but several parts of the resulting Home Assistant UX are still too indirect.

`current_soc`, `target_soc`, and `charge_loss` are currently exposed as MQTT number controls that present as slider-style UI. That is inconvenient for values users typically want to type precisely.

`pricing_information`, `charger_command`, and `charger_state` are also currently dependent on external MQTT wiring. That creates unnecessary automation glue for integrations that already exist inside Home Assistant, especially for Energi Data Service pricing and the actual charger entities.

This FR should reduce manual bridging and make EVCC integrate more naturally with Home Assistant selectors and live entity updates.

## 3. Scope

### 3.1. Functional Requirements

- Change `charge_loss` from a slider-style control to an input-style control.
- Change `current_soc` from a slider-style control to an input-style control.
- Change `target_soc` from a slider-style control to an input-style control.
- Replace the current free-form `pricing_information` JSON input with a selector that targets a Home Assistant device or entity from the Energi Data Service integration.
- EVCC must listen for state and attribute changes from the selected Energi Data Service source and maintain internal price data without requiring an external MQTT publisher for pricing.
- EVCC must update its internal pricing window continuously for up to `72` hours depending on available actual and forecast data.
- The selected pricing source must expose the pricing data to EVCC in attribute form, including actual and forecast information when available.
- Replace the current `charger_command` MQTT control with a selector for the actual charger switch entity ID in Home Assistant.
- EVCC must turn the selected charger switch on and off itself instead of relying on an external automation to mirror `charger_command`.
- Replace the current `charger_state` MQTT source with a selector for the actual charger state sensor entity ID in Home Assistant.
- EVCC must listen for state changes from the selected charger state sensor and map the following values into EVCC runtime state:
  - `charging`
  - `disconnected`
  - `connected_finished_idle`
  - `connected_requesting_charge`

### 3.2. Non-functional Requirements

- The updated configuration UX must feel native in Home Assistant and reduce the need for external bridging automations.
- EVCC must keep its runtime behavior deterministic when selected entities are unavailable, misconfigured, or report unsupported states.
- Logging should clearly identify invalid entity selections, missing attributes, and unsupported charger states without flooding the log.

### 3.3. Assumptions and Dependencies

- The target pricing integration is the Home Assistant Energi Data Service custom integration at `https://github.com/MTrab/energidataservice`.
- That integration provides a Home Assistant sensor whose attributes include pricing arrays such as `raw_today` and `raw_tomorrow`, and may also expose forecast-related data.
- EVCC will need selective Home Assistant API integration again for the chosen pricing entity, charger switch entity, and charger state sensor entity.
- The exact selector type may depend on what Home Assistant add-on/app configuration supports for integration-specific entity selection.

## 4. Out of Scope

- General support for arbitrary pricing integrations beyond Energi Data Service in this FR.
- Broad redesign of all EVCC MQTT entities introduced in `2.0.0`.
- Automatic discovery of every charger integration or automatic mapping of vendor-specific charger states beyond the four listed values.

## 5. Configuration and Data Impact

- Add configuration fields for:
  - Energi Data Service pricing source selection
  - charger switch entity selection
  - charger state sensor entity selection
- Update the Home Assistant-facing configuration schema and translations to reflect input-style numeric fields for `current_soc`, `target_soc`, and `charge_loss`.
- Pricing data handling will move from external MQTT JSON publishing toward internal synchronization from the selected Home Assistant source.

## 6. Acceptance Criteria

- `current_soc`, `target_soc`, and `charge_loss` are presented as typed inputs instead of slider-style controls.
- EVCC configuration allows selection of an Energi Data Service pricing source.
- EVCC reads and maintains internal pricing data from the selected pricing source without requiring external MQTT pricing updates.
- EVCC configuration allows selection of the real charger control switch entity.
- EVCC turns the selected charger switch on and off itself during runtime decisions.
- EVCC configuration allows selection of the real charger state sensor entity.
- EVCC correctly interprets the supported charger state values and updates runtime behavior accordingly.
- Automated tests cover selector-driven pricing updates, charger switch control, charger state mapping, invalid source handling, and the updated numeric input behavior.

## 7. Implementation Notes

- This FR intentionally revisits part of the `2.0.0` runtime model by reintroducing direct Home Assistant integration points for pricing and charger entities.
- The current EVCC runtime already contains a dormant Home Assistant API client module, which may be reused for the selected entity lookups and switch service calls instead of restoring the full pre-`2.0.0` entity-ID model.
- Home Assistant MQTT number entities likely need their UI mode changed so they render as typed inputs rather than sliders, if EVCC keeps those values on the MQTT device surface.
- Energi Data Service currently documents a sensor with attributes including `raw_today` and `raw_tomorrow`; implementation should align with those published attributes and only infer forecast handling where the selected source actually provides it.

## 8. Test Plan

- Add tests verifying `current_soc`, `target_soc`, and `charge_loss` discovery payloads render as input-style controls.
- Add tests for Energi Data Service source selection, attribute parsing, and live pricing refresh when attributes change.
- Add tests for actual-plus-forecast price window assembly across the next `24` to `72` hours depending on available data.
- Add tests for charger switch entity control through the Home Assistant API client.
- Add tests for charger state sensor mapping from:
  - `charging`
  - `disconnected`
  - `connected_finished_idle`
  - `connected_requesting_charge`
- Add tests for unsupported charger state values and unavailable selected entities.

## 9. Manual QA Checklist

- Configure EVCC with typed numeric inputs for `current_soc`, `target_soc`, and `charge_loss` and confirm they no longer render as sliders.
- Select an Energi Data Service pricing source and confirm EVCC imports prices from its attributes.
- Change the pricing source attributes in Home Assistant and confirm EVCC refreshes its internal price view without external MQTT pricing messages.
- Select the real charger switch and confirm EVCC can turn it on and off.
- Select the charger state sensor and confirm EVCC responds correctly to each supported state value.

## 10. Deployment and Rollback Notes

- This FR changes the post-`2.0.0` integration model by reintroducing selected Home Assistant entity dependencies.
- Release notes must explain which values remain MQTT-owned and which now come from Home Assistant selectors and listeners.
- Rollback should restore the previous `2.0.0` MQTT-bridged behavior for pricing and charger integration.
