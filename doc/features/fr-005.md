## 1. Summary

Implement the charging execution state machine defined in [state-machine.md](/c:/Users/AlexRasmussen/src/ev-charge-control/doc/misc/state-machine.md).
This feature will make EVCC apply explicit runtime control rules based on cable connection, authorization state, charging state, SoC state, and charge-window state.

## 2. Problem Statement

EVCC currently applies charging logic through a smaller set of runtime rules centered on schedule timing, authorization, charger state, and SoC.
The documented state machine is richer and includes explicit handling for cable connection, in-window versus past-window behavior, missed charging windows, unplugged charging states, and automatic reset when target SoC is reached.
Without implementing this state machine, EVCC behavior remains incomplete relative to the intended runtime model.

## 3. Scope

### 3.1. Functional Requirements

- Implement the state machine described in `doc/misc/state-machine.md`.
- Add a new required configuration input for an entity that indicates whether the EV cable is physically connected.
- Derive and evaluate these observed runtime states on each minute tick:
  - `Cable`
  - `Authorized`
  - `Charging`
  - `SoC`
  - `Charge Window`
- Implement the explicit transition rules from the written spec:
  - waiting before window while plugged and authorized
  - start charging when entering the window while plugged and authorized
  - warn when charging continues past the estimated window
  - alert and start charging when the window has been missed but authorization still exists
  - warn when in-window but authorization is missing
  - stop charging when unplugged while charging
  - auto-reset authorization and charging when SoC is reached
- Treat all combinations not listed in the state-machine spec as no-op transitions.
- Preserve the documented rule that row 10 is an unconditional auto-reset.
- Reflect the resulting status and runtime state in the published MQTT output.

### 3.2. Non-functional Requirements

- Keep the state-machine implementation explicit and testable rather than embedding it as ad hoc conditional logic.
- Preserve the current minute-tick scheduling model and quarter-hour calculation cadence unless the state machine explicitly changes a runtime action.
- Keep the distinction between observed state and commanded state clear in code.
- Avoid introducing behavior that is not specified in `doc/misc/state-machine.md`.

### 3.3. Assumptions and Dependencies

- The state-machine specification in `doc/misc/state-machine.md` is the governing behavior for this feature.
- `Charge Window` is derived from calculated `start` and estimated `end`, not `finish_by`.
- A new cable-connected entity will be needed because the current runtime model does not yet expose `Plugged | Unplugged`.
- The MQTT output path introduced in `fr-004` remains the publication mechanism for status and runtime output.

## 4. Out of Scope

- Replacing MQTT with a different output protocol.
- Changing the spreadsheet-derived state machine rules beyond what is already documented in `state-machine.md`.
- Adding fallback transitions for unspecified combinations.
- Persisting state-machine history or audit logs beyond the current output payload and normal logging.

## 5. Configuration and Data Impact

- Add a new required configuration option for a cable-connected entity.
- Add translation text for the new configuration field on the add-on config page.
- Extend the runtime model and published MQTT attributes to expose the derived cable/window/state-machine status as needed.
- Existing EVCC users will need to configure the new cable input before the feature can operate correctly.

## 6. Acceptance Criteria

- EVCC reads a configured cable-connected input and derives `Plugged | Unplugged`.
- EVCC evaluates the state machine on each minute tick using the documented precedence:
  1. auto-reset on `SoC = Reached`
  2. explicit listed transitions
  3. no-op default
- Each spreadsheet-derived rule in `state-machine.md` is implemented exactly once in code.
- Unlisted combinations do not change authorization or charging state.
- The resulting status and runtime lock behavior are reflected consistently in MQTT output.
- Automated tests cover all listed transition rules and the no-op default.

## 7. Implementation Notes

- Introduce a dedicated state-machine evaluation layer rather than further expanding `process_minute_tick` with inline branching.
- Model observed state separately from commanded actions so the implementation matches the written specification.
- Reuse the existing schedule timing and output publishing flow where possible, but let the state machine decide whether EVCC should command authorization or charging changes.
- The implementation should explicitly encode `Charge Window` as `Not Reached | In Window | Past Window` using the current schedule payload.
- The existing runtime lock logic will likely need to be replaced or absorbed into the state-machine action model.

## 8. Test Plan

- Add unit tests for deriving:
  - `SoC`
  - `Charge Window`
  - `Cable`
- Add state-machine tests covering every explicit transition rule from `state-machine.md`.
- Add tests proving row 10 auto-reset takes precedence over all other conditions.
- Add tests proving unspecified combinations are true no-op transitions.
- Add integration-style app tests verifying the resulting charging commands, authorization changes, status, and published MQTT output.

## 9. Manual QA Checklist

- Configure a valid cable-connected entity in Home Assistant.

- Verify EVCC starts charging with `ALERT` when the window has been missed but authorization remains.
- Verify EVCC stops charging if the cable is unplugged while charging.
- Verify EVCC auto-resets authorization and charging when target SoC is reached.
- Verify unlisted state combinations do not alter authorization or charging state.

## 10. Deployment and Rollback Notes

- Deployment requires the new cable-connected configuration to be set before the feature behaves correctly.
- MQTT consumers should be informed if additional state-machine-derived output fields are added.
- Rollback is straightforward, but any automations relying on the new state-machine behavior or cable input will need to be adjusted.
