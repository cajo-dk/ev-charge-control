## 0. Feature Request

- REQUEST ID: fr-008
- DATE: 2026.04.13
- STATUS: Implemented
- RELATED VERSION: 2.0.0
- IMPLEMENTATION BRANCH: fr-008

## 1. Summary

Reworked EVCC from a Home Assistant entity-ID driven runtime into an MQTT-owned runtime model. EVCC now exposes its operational inputs, outputs, and control surface as MQTT discovery entities on the EVCC device while preserving the aggregate MQTT payload for compatibility.

## 2. Problem Statement

The prior runtime depended on add-on configuration fields that pointed at Home Assistant helpers, sensors, and switches. That made EVCC configuration heavier, tied the runtime to Home Assistant API reads and writes, and split the operator experience across the add-on page and unrelated helper entities.

The new release consolidates the runtime interface onto the EVCC MQTT device so Home Assistant can interact with EVCC through one coherent surface.

## 3. Scope

### 3.1. Functional Requirements

- Keep only MQTT broker settings and log level in add-on configuration.
- Expose writable MQTT controls for EVCC runtime inputs.
- Expose read-only MQTT sensors for EVCC calculated outputs and operator status.
- Preserve the aggregate MQTT compatibility sensor.
- Add a `Start` MQTT button that re-enables authorization and starts charging immediately when already in-window.
- Introduce `status_message` and `status_level` with minute-by-minute updates and error precedence.

### 3.2. Non-functional Requirements

- Preserve the existing calculation and state-machine quality bar from Release 1.
- Keep MQTT topics retained so Home Assistant restores entity state cleanly after reconnect.
- Maintain automated test coverage for calculation, state-machine behavior, MQTT discovery, and MQTT message handling.

### 3.3. Assumptions and Dependencies

- Home Assistant automations or integrations feed live values into EVCC MQTT controls.
- Home Assistant or another MQTT producer mirrors real charger state into the EVCC `charger_state` MQTT sensor topic.
- External automation reacts to the EVCC `charger_command` state and forwards it to the physical charger integration.

## 4. Out of Scope

- Automatic discovery of external charger or car integrations.
- Automatic creation of Home Assistant automations that bridge real-world integrations to EVCC topics.
- Replacing MQTT as the EVCC integration surface.

## 5. Configuration and Data Impact

The add-on configuration schema is reduced to MQTT connectivity and log level only. Existing installs that relied on Home Assistant entity IDs in add-on options must migrate those values to MQTT-managed controls instead.

No persistent on-disk migration is required inside the add-on itself.

## 6. Acceptance Criteria

- EVCC add-on options contain only MQTT connectivity fields and `log_level`.
- EVCC publishes MQTT discovery payloads for the new controls, sensors, and Start button.
- EVCC consumes retained MQTT control values as runtime inputs.
- EVCC keeps publishing the aggregate MQTT compatibility sensor.
- EVCC publishes `status_message` and `status_level` according to the release rules.
- Automated tests cover MQTT discovery, control parsing, runtime ticks, Start button behavior, and status precedence.

## 7. Implementation Notes

- The Home Assistant API module remains in the repository, but the active runtime path no longer depends on it.
- `charger_state` is modeled as an observed read-only MQTT sensor while `charger_command` remains EVCC's desired charger output.
- Pricing input is supplied as a single JSON text payload matching the existing `raw_today` / `raw_tomorrow` / `forecast` structure.

## 8. Test Plan

- Verify MQTT discovery for the aggregate sensor, runtime controls, read-only sensors, and Start button.
- Verify runtime store validation for numeric, switch, time, and pricing payloads.
- Verify status transitions for waiting, active charging, completed charging, disabled automation, ready state, and errors.
- Verify Start button behavior and charger-command publication when the schedule is already in-window.

## 9. Manual QA Checklist

- Configure the add-on with only MQTT broker settings.
- Confirm Home Assistant creates the EVCC MQTT entities after startup.
- Publish representative runtime inputs and pricing JSON to the EVCC control topics and confirm EVCC calculates a schedule.
- Press Start in Home Assistant and verify authorization is enabled and charging starts immediately when already in-window.
- Confirm `status_message` and `status_level` move through waiting, active, completed, ready, and error states.

## 10. Deployment and Rollback Notes

This change is released as `2.0.0` because it breaks the previous add-on configuration model. Upgrade notes must direct users to migrate from add-on entity-ID settings to MQTT-driven controls before relying on EVCC scheduling again.

Rollback requires reverting to the latest `1.7.x` release and restoring the previous add-on options.
