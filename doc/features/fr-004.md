## 0. Feature Request

- REQUEST ID: fr-004
- DATE: 2026.03.14
- STATUS: Released
- RELATED VERSION: 1.4.0
- IMPLEMENTATION BRANCH: fr-004

## 1. Summary

Replace the current `input_text` JSON output with an MQTT-published Home Assistant entity.
The published EVCC entity must expose `status` as its state and all other EVCC output fields as attributes so the data can be consumed natively in Home Assistant dashboards, templates, and automations.

## 2. Problem Statement

EVCC currently writes a JSON string into an `input_text` helper.
That forces Home Assistant consumers to parse JSON even for simple uses such as checking status, start time, charger state, or SoC values.
Home Assistant helpers do not provide a supported way for EVCC to write custom attributes through the current API path.
Using MQTT would allow EVCC to publish a proper Home Assistant entity with a native state/attributes model.

## 3. Scope

### 3.1. Functional Requirements

- Publish the EVCC output over MQTT instead of writing it to an `input_text` helper.
- Publish `status` as the MQTT entity state.
- Publish all remaining EVCC output properties as MQTT entity attributes.
- Preserve the current information content already emitted by EVCC, including the fields added in `fr-003`.
- Keep the published state and attributes updated for:
  - successful calculation writes;
  - runtime execution-state writes; and
  - error writes.
- Publish the EVCC output as a Home Assistant MQTT entity discoverable by Home Assistant.
- Continue updating the published entity whenever EVCC currently rewrites its result helper payload.

### 3.2. Non-functional Requirements

- The output model must fit Home Assistant naturally and remain reliable across add-on restarts.
- The implementation must not depend on unsupported writes to helper attributes.
- MQTT publishing must handle reconnects and broker unavailability without crashing EVCC.
- The published payload contract must be documented clearly so users can debug or consume it outside Home Assistant if needed.
- Existing automation consumers will need a migration path because the output contract changes materially.

### 3.3. Assumptions and Dependencies

- EVCC will use MQTT as the new output protocol for this feature.
- The current output target is an `input_text` helper configured in the add-on options and will be replaced or superseded by MQTT output configuration.
- Home Assistant will consume the EVCC output through MQTT discovery or a documented equivalent MQTT entity pattern.
- An MQTT broker must be available to both EVCC and Home Assistant.

## 4. Out of Scope

- Redesigning the charge calculation logic.
- Changing the meaning of any existing output fields.
- Adding new output fields beyond the already approved payload content.
- Replacing other Home Assistant API reads or charger-control actions with MQTT in this FR.

## 5. Configuration and Data Impact

- Add MQTT output configuration required for publishing the EVCC entity.
- Replace or deprecate the current `result_helper_entity` output configuration.
- Existing dashboards, automations, and templates that read the JSON string will need to be updated.
- Existing Home Assistant setups will need MQTT broker configuration if not already present.

## 6. Acceptance Criteria

- Home Assistant can read `status` directly as MQTT entity state.
- All remaining EVCC output fields are available as attributes on the published MQTT entity.
- Successful, runtime-state, and error updates all publish consistent state/attribute data.
- EVCC handles temporary MQTT publish failures without terminating the service.
- The implementation approach is documented clearly enough for users to migrate from the existing JSON helper output.

## 7. Implementation Notes

- Introduce a dedicated MQTT publishing layer instead of overloading the Home Assistant API client.
- Publish a Home Assistant MQTT-discoverable entity whose state topic carries `status` and whose attributes topic carries the remaining EVCC output payload.
- Keep EVCC's internal output model centralized so calculation, execution, and error writes all produce the same state/attribute publication data.
- Document the migration from `input_text` output to MQTT entity output in the release notes and README.

## 8. Test Plan

- Add tests for the MQTT publishing client.
- Add coverage for state publishing and attribute publishing on success and error paths.
- Add coverage for MQTT-discovery payload generation if discovery is used.
- Add migration-oriented validation if the output configuration model changes.

## 9. Manual QA Checklist

- Verify Home Assistant creates or updates the EVCC MQTT entity.
- Verify the published entity shows `status` as its state in Home Assistant.
- Verify start time, end time, finish-by, charger state, authorization state, and SoC values appear as attributes.
- Verify an error condition updates state and attributes consistently.
- Verify EVCC resumes publishing correctly after an MQTT broker reconnect.

## 10. Deployment and Rollback Notes

- This feature requires MQTT broker availability and EVCC MQTT configuration before rollout.
- Home Assistant consumers of the existing JSON payload will need to migrate to the MQTT entity before or during rollout.
- Rollback will require restoring the previous output configuration and switching consumers back to the legacy JSON helper output if that path is retained temporarily.
